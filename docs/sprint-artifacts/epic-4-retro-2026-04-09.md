## Successes and Strengths

1. **4th consecutive 100% completion with 0 production incidents** — Delivery consistency is now a firmly established pattern across the project.

2. **SAVEPOINT isolation pattern discovery** — Code review on Story 4-1 uncovered a critical session poisoning bug where `db.flush()` failure inside `AuditService.log()` left the SQLAlchemy session in `PendingRollbackError` state. Fixed with `db.begin_nested()` (SAVEPOINT). This is now permanent infrastructure and a reusable pattern.

3. **Story intelligence compounding continues** — Fourth epic where "Previous Story Intelligence" sections carry actionable learnings forward. Error conventions, auth test patterns, and pre-review checklist from previous retros were all embedded in Epic 4 story specs.

4. **Graceful degradation by design** — The fire-and-forget architecture means audit failures can never break business operations. Worst case is a missing audit record, never a failed admin operation. Layered defense: SAVEPOINT in service, rollback in wrapper, caller's data already committed.

5. **Clean `_audit_log` helper pattern** — Single fire-and-forget wrapper used 13 times across two router files. DRY without being over-abstracted.

6. **46 new tests with 100% coverage** — Clean split between infrastructure tests (4-1) and integration tests (4-2).

## Challenges and Growth Areas

1. **Session poisoning bug was high-severity, caught only by adversarial review** — First high-severity finding since Epic 1. The fire-and-forget pattern sounded simple but had subtle session state implications. Unit tests all passed — the bug only manifested when a flush failed mid-transaction with a real caller. Reinforces the value of adversarial code review.

2. **Edge-case tests defaulted to service level** — Both stories had code review findings for missing HTTP-level tests on edge cases (date range filters with Query aliases, ordering through the endpoint, no-change scenarios). Pattern: happy path tested through HTTP, edge cases only at unit level.

3. **Dead code shipped from spec contradiction** — `AuditLogFilter` schema was created per story spec but never referenced by any endpoint (which used inline Query params). Caught in review round 2. The spec had conflicting instructions.

4. **pytest-mock dependency assumption** — Story 4-2 tests assumed `mocker` fixture was available, but `pytest-mock` wasn't installed. Pivoted to `unittest.mock.patch` context managers. Minor but cost time.

5. **Growing frontend anxiety** — 4 epics of backend-only work means the frontend is untouched and has 23 broken tests. Epic 5 (6 frontend stories) will be a significant gear shift.

---

## Key Insights

1. **"Test that it doesn't raise" ≠ "test what state it leaves behind"** — The fire-and-forget tests verified exceptions didn't propagate, but didn't test the session state after failure. The session poisoning test added after review tests the actual state. Both perspectives are needed.

2. **Edge cases involving query params and serialization need HTTP-level tests** — Service-level tests don't cover Query aliases, response serialization, or pagination through the endpoint. When an edge case touches the HTTP layer, test it there.

3. **Story specs can have bugs** — When a spec says "create X" in one task but never references X in any endpoint, the dev should flag the contradiction rather than implementing both sides. Dead code is a spec bug, not a dev oversight.

4. **Fire-and-forget is architecturally simple but implementation-subtle** — The session state interaction between `db.flush()`, exception handling, and the caller's transaction required SAVEPOINT isolation. Simple patterns can hide complex failure modes.

## Team Agreements

1. **Edge-case HTTP testing rule:** Edge cases involving query params, response serialization, or aliases get at least one HTTP-level test, not just service-level.

2. **Spec contradiction protocol:** Devs flag story spec contradictions rather than implementing both sides. Dead code from spec bugs is avoidable.

3. **Pre-implementation policy (carried from Epic 3):** Allowed when code is naturally cohesive, with full independent code review on the subsequent story.
